Flow 1: Generating a Training Asset

A user asks Canon to produce a facilitator guide for teaching Tool Use to enterprise developers.

canon generate \
  --type guide \
  --subtype setup \
  --concepts "tool_use,system_prompt" \
  --audience enterprise_developer \
  --format facilitator_guide
1
Resolve the Ontology Subgraph
Canon walks the knowledge graph from the requested concepts outward. Starting from tool_use and system_prompt, it collects:
• Capabilities they support (structured data extraction, conversation control)
• Tasks those capabilities enable (configure tool definitions, set system instructions)
• Prerequisites (API authentication — you need an API key before you can use tools)
• Constraints (rate limits, token limits, tool schema validation rules)
• Evidence sources linked to each concept
2
Retrieve Evidence via RAG
For each evidence source in the subgraph, Canon retrieves relevant content. Evidence objects in the ontology point to source material — documentation pages, API references, internal notes. RAG searches within those sources using the concept and task descriptions as queries, returning specific passages.

The ontology scopes the RAG search: instead of searching all documents for "tool use," Canon only searches the sources explicitly linked to the tool_use concept. This prevents hallucination from unrelated content.
3
Assemble the Generation Prompt
Canon builds a structured prompt for the LLM containing:
• The asset template (facilitator guide structure: overview, prerequisites, steps, validation, troubleshooting)
• The resolved ontology subgraph (concepts, relationships, constraints)
• Retrieved evidence passages from RAG
• Audience profile (enterprise developer — assumes API familiarity, prefers code examples)
• Generation rules (must cite evidence, must not make claims beyond sources, must flag uncertainty)
4
LLM Generates the Draft Asset
The LLM (Claude or OpenAI, depending on configured API key) generates the facilitator guide. The output is structured — not free-form prose — with sections mapped to the template.

Dry-run mode: Instead of calling an LLM, Canon assembles the asset from pre-written content blocks stored alongside each knowledge object. Each concept, task, and capability can have a content_block field — a human-written explanation that gets composed into the asset template. No API key needed.
5
Validate & Score
Canon checks the generated asset against ontology constraints:
• Does it reference all required concepts?
• Are evidence links present for factual claims?
• Are prerequisites listed?
• Does it respect constraints (e.g., doesn't claim unlimited context window)?

6
Store the Asset in the Graph
The generated asset is saved as a new node in the knowledge graph with full metadata:
• Type, subtype, format, audience
• Links to source concepts, capabilities, tasks, evidence
• Confidence scores per dimension
• Lifecycle state: draft (AI-generated assets always start as drafts)
• Generation timestamp and method (LLM vs dry-run)
Result
A structured facilitator guide exists in the graph, linked to every knowledge object it was built from. Any change to those objects is traceable back to this asset.

Flow 2: Source Material Changes (Drift Detection)

Anthropic updates their documentation: the context window for Claude increases from 200K to 500K tokens. Here's what happens.

canon drift \
  --evidence context_window_docs \
  --change "Max context window increased from 200K to 500K tokens" \
  --new-version "2026-03-10"
1
Record the Evidence Change
The evidence object context_window_docs is updated:
• last_verified date changes to the new version date
• A change description is recorded: "Max context window increased from 200K to 500K tokens"
• The evidence is marked as updated, triggering downstream propagation
2
Trace the Impact Through the Graph
Canon walks the graph from the changed evidence outward:

context_window_docs
  └─ verifies → Context Window (concept)
      └─ supports → Long Document Processing (capability)
      └─ supports → Multi-turn Conversation (capability)
      └─ taught by → 3 assets:
          • Prompt Engineering Facilitator Guide
          • Context Window Best Practices (reference)
          • Long Document Processing Demo
3
Recalculate Confidence Scores
Each affected asset has its confidence recalculated. The freshness dimension drops because the asset was generated against old evidence. The evidence dimension drops because the asset now contains a factual claim ("200K context window") that contradicts the updated source.

4
Flag for Review & Generate Update Proposal
Assets whose confidence drops below the threshold (e.g., 0.70) are moved to needs_review lifecycle state.

Canon generates an update proposal for each affected asset: 
{
  "asset": "Prompt Engineering Facilitator Guide",
  "reason": "Evidence changed: context_window_docs",
  "stale_claims": [
    {
      "section": "Context Window Overview",
      "current_text": "Claude supports up to 200K tokens...",
      "evidence_says": "Max context window increased to 500K tokens",
      "suggested_fix": "Update token limit reference from 200K to 500K"
    }
  ],
  "affected_sections": [
    "Context Window Overview",
    "Long Document Processing Exercise",
    "Troubleshooting: Token Limit Errors"
  ],
  "proposed_action": "regenerate_sections",
  "requires_human_review": true
}
5
Apply Updates (with approval)
A reviewer (or in demo mode, the user) approves the proposal. Canon regenerates only the affected sections — not the entire asset — using the updated evidence. The new sections go through the same validation and scoring pipeline from Flow 1.

After update:
• Confidence scores recover
• Asset returns to reviewed lifecycle state
• An update record is logged (what changed, why, when, who approved)
6
Cascade Check
Canon checks if the updated asset is referenced by other assets (e.g., a workshop agenda that includes the facilitator guide). If so, those downstream assets also have their confidence recalculated. The cascade continues until no more assets are affected.
The mechanism
Drift detection doesn't require monitoring live URLs (though it could, later). The trigger is explicit: "this evidence changed." The system's value is in knowing what's affected — tracing from one changed fact through the ontology to every asset that depends on it — and proposing specific fixes rather than just flagging "something is stale."

How RAG Fits the Ontology

Key Idea
The ontology scopes the RAG. Instead of "search everything for tool use," Canon says "search only the evidence sources linked to the tool_use concept." This makes retrieval precise and prevents the LLM from pulling in unrelated content.

Evidence objects in the ontology store metadata (URL, last verified date, version) and point to the actual source content stored in the RAG index. When Canon needs to generate or validate an asset, it uses the ontology to determine which evidence to retrieve, then uses RAG to get the specific passages relevant to the current task.

LLM Configuration

Canon supports three authentication modes for LLM access:

Anthropic API Key
Set ANTHROPIC_API_KEY env var. Uses Claude via api.anthropic.com.
OpenAI API Key
Set OPENAI_API_KEY env var. Uses GPT-4o or similar.
Claude OAuth
Authenticate via Claude.ai account (Pro/Max). Uses OAuth 2.0 + PKCE flow. Token is a standard Anthropic API key (sk-ant-oat) used as Bearer token.
Dry Run (no API key)
Assets assembled from pre-written content blocks on each knowledge object. No LLM call. Useful for demo and testing.

CLI Commands (Proposed)

canon graph                          # browse the knowledge graph
canon graph --concept tool_use       # show a concept and its connections
canon graph --gaps                   # find coverage gaps (concepts without assets, etc.)

canon generate \                     # generate an asset from knowledge objects
  --type guide \
  --concepts "tool_use" \
  --audience enterprise_developer \
  --dry-run                          # use pre-written blocks, no LLM

canon drift \                        # report a source change
  --evidence context_window_docs \
  --change "context window increased to 500K"

canon status                         # overall health: confidence scores, review queue
canon status --asset "Prompt Eng..." # detailed status for one asset

canon review                         # see assets needing review
canon review --approve <asset_id>    # approve an update proposal

canon adapt \                        # re-target an asset for a new audience
  --asset "Tool Use Guide" \
  --audience support_engineer
